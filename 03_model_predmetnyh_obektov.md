## Назначение

Настоящий документ определяет требования к проектированию предметных объектов продукта.

Цель документа — обеспечить совместимость модели данных с будущим внедрением FSM Engine без изменения структуры основных сущностей.

## Общий принцип

Предметная сущность должна хранить факты, а не бизнес-логику.

Например: `Order` хранит дату создания, покупателя, продавцов, товары, суммы. `Order` не определяет: можно ли подтвердить заказ; можно ли начать сборку; можно ли завершить заказ. Эти решения принимает Domain Service.

## Требование 1. Сущности являются пассивными

Предметные объекты содержат данные, не принимают бизнес-решений.

Например: `PurchaseIntent` содержит товары, количество, ограничения, предпочтения — но не содержит методов `start()`, `finish()`, `approve()`.

## Требование 2. Отсутствие вычисляемых состояний

Сущность не должна хранить вычисляемое состояние, если оно может быть получено из других данных.

Не рекомендуется хранить `isCompleted`, `isReady`, `isAvailable`, если эти признаки вычисляются.

## Требование 3. Статусы являются временным механизмом

До внедрения FSM Engine допускается использование технического поля `status`. Однако: status не является источником бизнес-логики; status используется только для совместимости текущей реализации; после внедрения FSM Engine статус может стать производным значением.

## Требование 4. Идентификаторы неизменяемы

Все основные сущности имеют постоянный идентификатор (например `BasketId`, `PurchaseIntentId`, `SellerId`, `ProductId`). Идентификатор не зависит от состояния объекта.

## Требование 5. История изменений отделяется от текущего состояния

Каждая сущность описывает только своё текущее состояние. История изменений хранится отдельно. Например: `PurchaseIntent` / `PurchaseIntentHistory`, или `Catalog` / `CatalogChanges`.

## Требование 6. Ссылочная целостность

Сущности связываются только идентификаторами. Не допускается дублирование данных между сущностями. Например: `PurchaseOption` содержит `SellerId`, информация о продавце хранится только в `Seller`.

## Требование 7. Независимость от UI

Сущности не содержат: цветов; иконок; названий кнопок; элементов интерфейса; настроек отображения. Эти данные принадлежат ViewModel.

## Требование 8. Независимость от API

Сущности не проектируются под конкретный REST-запрос. REST собирает ViewModel из предметных объектов.

## Требование 9. Независимость от FSM Engine

Предметные объекты не знают: о переходах; о графе состояний; о правилах переходов. FSM Engine работает поверх предметной модели.

## Требование 10. Расширяемость

Добавление новых возможностей не должно требовать изменения существующих предметных сущностей, если достаточно добавить новые связи и новые сервисы.

## Критерии приёмки

При проектировании каждой новой сущности проверяется: содержит ли она только данные; отсутствует ли в ней бизнес-логика; отсутствует ли зависимость от UI; отсутствует ли зависимость от REST API; отсутствует ли зависимость от FSM Engine; может ли она использоваться как при CRUD, так и при FSM.

---

**Комментарий из чата:** до сих пор архитектура готовилась сверху вниз (UI → Platform Core → сервисы). Теперь она закрепляется снизу вверх. Если предметные сущности останутся «чистыми», то внедрение FSM Engine действительно сведётся к замене механизма принятия решений, а не к переделке модели данных. Это также даёт разработчику импорта каталога чёткие правила проектирования БД и доменных объектов без привязки к текущей реализации бизнес-процессов.
