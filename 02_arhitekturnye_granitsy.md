# Архитектурные границы (Client UI / Platform Core / REST / Application / Domain / Repository / DB)

> Текст оставлен как получен, в исходных примерах упоминается GreenMarket — правила ниже одинаково применимы к любому продукту на Platform Core.

**Вводная реплика чата:** дальше нужно закрепить последнее слабое место — описаны сервисы, но не границы ответственности между слоями. Именно здесь обычно начинается «утекание» логики в React, Controller или Repository.

## Назначение

Настоящий документ определяет ответственность каждого архитектурного слоя продукта на Platform Core.

Ни один слой не должен выполнять функции другого слоя. Настоящее правило является обязательным для всех разработчиков проекта.

## Общая архитектура

```
Client UI
  ↓
Platform Core
  ↓
REST API
  ↓
Application Layer
  ↓
Domain Services
  ↓
Business Process Engine (будущая реализация)
  ↓
Repositories
  ↓
Database
```

## Client UI

Отвечает только за отображение.

Разрешается: отображать ViewModel; отправлять Action; получать обновлённую ViewModel.

Запрещается: принимать бизнес-решения; вычислять доступные действия; изменять сущности предметной области; обращаться к БД; обращаться к Repository.

## Platform Core

Отвечает только за взаимодействие.

Разрешается: принимать Action; доставлять Action; публиковать Events; синхронизировать каналы взаимодействия.

Запрещается: знать бизнес конкретного продукта; принимать решения предметной области.

## REST API

Отвечает только за транспорт.

Разрешается: принимать запросы; проверять формат; вызывать Application Layer.

Запрещается: реализовывать бизнес-логику; обращаться напрямую к Repository.

## Application Layer

Координирует выполнение операций.

Разрешается: запускать Domain Services; объединять несколько сервисов; управлять транзакциями.

Запрещается: принимать бизнес-решения.

## Domain Services

Единственное место бизнес-логики.

Разрешается: принимать решения; применять бизнес-правила; проверять ограничения; создавать Business Events.

Запрещается: знать React; знать REST; знать Platform Core; знать способ хранения данных.

## Business Process Engine

После внедрения становится внутренним механизмом Domain Services.

Business Process Engine определяет допустимые операции над бизнес-процессом, вычисляет результат и генерирует Business Events. Если Business Process Engine реализован на основе FSM — он также определяет допустимые переходы и вычисляет состояние.

Business Process Engine не взаимодействует напрямую с UI или REST API.

## Repository

Отвечает исключительно за хранение данных.

Разрешается: загрузить; сохранить; удалить; выполнить поиск.

Запрещается: вычислять состояние; принимать решения; публиковать события; выполнять операции Business Process Engine.

## Database

Хранит данные. Не содержит бизнес-логики.

## Правило зависимостей

Каждый слой знает только следующий слой: `UI → Platform Core → REST → Application → Domain → Repository → DB`. Обратные зависимости запрещены.

## Запрещённые зависимости

- Client UI → Repository
- Client UI → Database
- REST → Repository
- Repository → Platform Core
- Repository → Domain Service
- Business Process Engine → UI
- Business Process Engine → React
- Business Process Engine → REST

## Критерии приёмки

При код-ревью проверяется: UI не содержит бизнес-логики; REST не содержит бизнес-логики; Repository не содержит бизнес-логики; Domain Services являются единственным местом принятия решений; Business Process Engine (после внедрения) используется только внутри Domain Services.

---

**Комментарий из чата:** здесь заканчивается переход от требований «что делать» к требованиям «где это должно находиться» — критично, потому что над проектом уже работают несколько человек. Если заранее не закрепить архитектурные границы, через несколько недель логика неизбежно начнёт расползаться по React, контроллерам и репозиториям, а внедрение Business Process Engine (в любой реализации — CRUD, FSM или иной) действительно станет сложным.
