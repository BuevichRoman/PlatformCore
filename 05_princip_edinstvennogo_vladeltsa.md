# Принцип единственного владельца бизнес-решения

**Вводная реплика чата:** документ нужен не программисту, а архитектору. Он закрепит один принцип, который потом избавит от многих споров.

## Назначение

Настоящий документ определяет, где в системе принимаются бизнес-решения.

Для каждого бизнес-решения существует только один ответственный компонент. Дублирование бизнес-логики запрещается.

## Основной принцип

Каждое правило предметной области реализуется только в одном месте.

Например, вопрос «Можно ли начать покупку?» должен иметь один ответ. Не допускается: одна проверка в React; другая в REST; третья в Service; четвёртая в FSM.

## Правило 1. Client UI никогда не принимает бизнес-решений

Client UI: отображает информацию; отправляет Action; показывает результат.

UI не отвечает на вопросы: можно ли выполнить действие; какое состояние наступит; какой вариант лучше.

## Правило 2. REST API не принимает бизнес-решений

REST: принимает запрос; проверяет формат; передаёт Action. REST не определяет: допустимость операции; переходы; ограничения.

## Правило 3. Repository не принимает бизнес-решений

Repository отвечает только за хранение данных. Repository никогда не отвечает: можно ли изменить объект; какой переход допустим; какие события публиковать.

## Правило 4. Domain Service является владельцем бизнес-решений

До внедрения FSM Engine именно Domain Service отвечает: можно ли выполнить Action; какие объекты изменятся; какие события возникнут.

## Правило 5. После внедрения FSM Engine владельцем переходов становится FSM Engine

FSM Engine отвечает только за: допустимость переходов; вычисление нового состояния; генерацию переходов. FSM Engine не отвечает: за REST; за UI; за хранение данных.

## Правило 6. Business Rules существуют в единственном экземпляре

Если правило изменилось, оно изменяется только в одном месте. Запрещается копировать одинаковые проверки между сервисами.

**Пример.** Правило: «Покупку нельзя завершить, пока не обработаны все обязательные товары».

Правильно: проверка реализована один раз в соответствующем Domain Service (или позже — в FSM Engine).

Неправильно: UI скрывает кнопку; REST возвращает ошибку; Service проверяет повторно; FSM содержит ещё одну проверку.

## Правило 7. UI получает результат, а не вычисляет его

Вместо `if (basket.items == 0)` UI получает `canStartPurchase = false`, `reason = "Basket is empty"`.

## Правило 8. При переносе логики в FSM Engine не меняется её владелец

Меняется только реализация. Внешние контракты остаются прежними.

## Критерии приёмки

Для каждого нового бизнес-правила в код-ревью должны быть даны ответы: где находится единственная реализация этого правила? есть ли копии этого правила в других слоях? можно ли удалить остальные проверки без изменения поведения системы?

Если ответ на второй вопрос «да», правило реализовано неверно.

---

**Комментарий из чата:** это тот документ, который через полгода сэкономит больше времени, чем любой другой. В проектах с несколькими разработчиками логика почти всегда начинает дублироваться: немного в UI, немного в контроллере, немного в сервисе. Потом при изменении правила одна проверка обновляется, другая забывается, и появляются трудноуловимые ошибки.

Если с самого начала зафиксировать принцип единственного владельца бизнес-решения, то переход на FSM Engine станет не переносом логики из пяти мест в одно, а простой заменой одного механизма реализации другим.
