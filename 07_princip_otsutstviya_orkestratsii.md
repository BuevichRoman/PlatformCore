## Назначение

Настоящий документ определяет правила построения бизнес-процессов продукта на Platform Core.

В системе не допускается существование компонентов, управляющих полным сценарием выполнения бизнес-процесса. Бизнес-процессы возникают как следствие последовательности независимых Action и Business Event.

## Основной принцип

В системе отсутствует центральный объект вида: `PurchaseWorkflow`, `PurchaseManager`, `ScenarioEngine`, `BusinessOrchestrator`.

Ни один компонент не должен содержать полный сценарий выполнения процесса.

## Правило 1. Каждый сервис знает только своё действие

Например: `PurchaseService` знает `AddProduct`, `RemoveProduct`, `FinishPurchase`; `CatalogService` знает `UpdateCatalog`, `UpdateStock`; `InventoryService` знает `UpdateHomeInventory`. Ни один сервис не знает полного сценария процесса.

## Правило 2. Следующее действие определяется событиями

Пример:

```
AddProduct
  ↓
ProductAdded
  ↓
PurchaseIntentChanged
  ↓
PurchaseOptimizer запускает перерасчёт
  ↓
PurchaseOptionsCalculated
  ↓
UI обновляется
```

Ни один сервис не вызывает следующий сервис напрямую.

## Правило 3. Сервисы взаимодействуют только через события

Запрещается: `PurchaseService → CatalogService → InventoryService → MapService`.

Правильно: `PurchaseService → Business Event →` (Catalog реагирует, Inventory реагирует, Map реагирует, Analytics реагирует).

## Правило 4. Любой сервис может быть удалён

Удаление любого сервиса не должно требовать переписывания остальных. Например, если завтра убрать `RecommendationService`, `PurchaseService` продолжит работать.

## Правило 5. Любой сервис может быть добавлен

Например, через год появляется `NutritionService`. Он подписывается на `PurchaseCompleted` и начинает строить рекомендации. Остальная система не изменяется.

## Правило 6. FSM Engine не становится оркестратором

После внедрения FSM Engine отвечает только за: допустимость Action; изменение состояния; генерацию Business Event. FSM Engine не управляет сервисами, не вызывает UI, не вызывает Repository, не запускает другие процессы.

## Правило 7. Platform Core не становится оркестратором

Platform Core отвечает только за: доставку Action; доставку Business Event; взаимодействие компонентов. Platform Core не принимает решений о порядке выполнения бизнес-процессов.

## Архитектурная схема

Правильно: `Action → Domain Service → Business Event → Подписчики`.

Неправильно: `Action → Workflow Manager → Service 1 → Service 2 → Service 3 → Service 4`.

## Критерии приёмки

Каждый новый сервис проверяется по вопросам: знает ли сервис полный сценарий процесса? вызывает ли он другие Domain Service напрямую? может ли он работать независимо? можно ли удалить его без изменения остальных сервисов?

Если хотя бы один ответ отрицательный, архитектура нарушена.

## Архитектурное следствие

После появления FSM Engine архитектура не изменится. Изменится только механизм принятия решений внутри Domain Service. Все взаимодействия между сервисами останутся событийными.

---

**Комментарий из чата:** здесь формулируется принцип шире, чем просто подготовка к FSM: в системе не должно появиться «дирижёра», который знает весь бизнес-процесс. Это особенно ценно для платформы в целом, потому что со временем будут добавляться новые подсистемы, и они смогут подключаться через события, не заставляя переписывать уже работающие сервисы. Такой подход хорошо согласуется с идеей Platform Core как универсального слоя взаимодействия, а не центра бизнес-логики.
